O que vamos fazer

Criar um ALB distribuindo tráfego entre instâncias EC2, configurar Auto Scaling para escalar automaticamente, monitorar métricas com CloudWatch e alertas via SNS, e gerenciar acessos com IAM.

Parte 1 — Load Balancer + Auto Scaling
Passo 1 — Criar um Modelo de execução (Launch Template)
01

Criar o Modelo de execução

O modelo que o Auto Scaling usará para criar novas instâncias

💡

O que é um Modelo de execução (Launch Template)? É um modelo que define as configurações de uma instância EC2 — AMI, tipo, chave, Security Group, User Data. O Auto Scaling usa esse modelo para criar novas instâncias automaticamente quando necessário.

  • Vá em EC2 → Modelo de execuçãos → Criar Modelo de execução
  • Nome: meu-template
  • AMI: Ubuntu Server 24.04 LTS
  • Tipo de instância: t3.small
  • Par de chaves: vockey
  • Security Group: selecione um grupo de segurança com permissão para SSH, HTTP e HTTPS ou crie um novo grupo de segurança e adicione as 3 regras (origem qualquer lugar)
  • Expanda Detalhes avançados → Dados do usuário e cole:
#!/bin/bash
apt update -y
apt install -y nginx
systemctl start nginx
systemctl enable nginx
# Página de identificação da instância
echo "<h1>Instância: $(hostname -f)</h1>" > /var/www/html/index.html
💡

O User Data instala o nginx e escreve o hostname da instância na página. Isso permite ver na prática o Load Balancer alternando entre instâncias diferentes durante os testes.

  • Clique em Criar Modelo de execução
Passo 2 — Criar o Grupo de destino
02

Criar o Grupo de Destino (Target Group)

O grupo de instâncias que receberá o tráfego do Load Balancer

💡

O que é um Grupo de destino? Um conjunto de destinos (instâncias EC2) que o Load Balancer pode encaminhar requisições. O ALB verifica periodicamente a saúde de cada instância e só envia tráfego para as saudáveis.

  • Vá em EC2 → Balanceamento de carga → Grupos de destino → Criar Grupo de destino
  • Tipo de destino: Instâncias
  • Nome: meu-target-group
  • Protocolo: HTTP | Porta: 80
  • VPC: deixe a VPC padrão
  • Verificações de integridade — protocolo: HTTP | Caminho: /
  • Clique em Próximo, Próximo e depois em Criar Grupo de destino
⚠️

Sem instâncias por enquanto — o Grupo de destino começa vazio. O Auto Scaling vai registrar as instâncias automaticamente mais à frente.

Passo 3 — Criar o Application Load Balancer (ALB)
03

Criar o ALB (Application Load Balancer)

O ponto de entrada único que distribui o tráfego entre as instâncias

💡

O que é um ALB? Um Load Balancer de camada 7 (HTTP/HTTPS) que recebe todo o tráfego de entrada e distribui entre as instâncias do Grupo de destino, usando algoritmo round-robin por padrão. Se uma instância cair, o ALB para de enviar tráfego para ela automaticamente.

  • Vá em EC2 → Balanceamento de carga → Criar load balancer
  • Em Application Load Balancer clique em Criar
  • Nome: meu-alb
  • Esquema: Voltado para a internet
  • Tipo de endereço IP: IPv4
  • VPC: selecione a VPC padrão
  • Zonas de disponibilidade: marque todas as AZs disponíveis
  • Security Group: crie um novo ou selecione um com porta 80 liberada para 0.0.0.0/0
  • Em Listeners: protocolo HTTP, porta 80, Grupo de destino, selecione meu-target-group
  • Clique em Criar load balancer
💡

Múltiplas AZs: ao selecionar todas as zonas de disponibilidade, o ALB pode distribuir tráfego entre instâncias em datacenters diferentes — garantindo alta disponibilidade mesmo se uma AZ tiver problemas.

Passo 4 — Criar o grupo Auto Scalling
04

Criar o Grupo Auto Scalling

Escalar instâncias automaticamente conforme a demanda

💡

O que é Auto Scaling? Um serviço que monitora a carga das suas instâncias e adiciona ou remove EC2s automaticamente dentro de limites que você define. Se a CPU subir muito, ele sobe mais instâncias. Se a carga cair, ele termina as desnecessárias para economizar custo.

  • Vá em EC2 → Grupo Auto Scalling → Criar Grupo Auto Scalling
  • Nome: meu-asg
  • Launch template: selecione meu-template
  • VPC: VPC padrão — selecione todas as sub-redes disponíveis (pelo menos as 4 primeiras)
  • Em Balanceamento de carga: selecione Anexar a um load balancer existente → meu-target-group
  • Ative a opção Verificações de integridade do Elastic Load Balancing depois clique em próximo
ConfiguraçãoValorSignificado
Capacidade desejada2Número inicial de instâncias
Capacidade mínima1Nunca desce abaixo de 1
Capacidade máxima4Nunca sobe acima de 4
  • Em Ajuste de escala automático: selecione Política de dimensionamento com monitoramento de objetivo
  • Métrica: Média de utilização de CPU | Valor destino: 60 (representa a porcentagem)
  • Clique em Próximo. Próximo. Próximo. Criar Grupo Auto Scalling
✅

O Auto Scaling vai criar 2 instâncias usando o Launch Template, registrá-las no Target Group e o ALB começará a distribuir o tráfego entre elas automaticamente.

🧪 Teste — Load Balancer em ação

  • Vá em EC2 → Load Balancers e copie o DNS name do meu-alb
  • Acesse http://<DNS-DO-ALB> no navegador várias vezes
  • O hostname na página deve alternar entre as duas instâncias — confirmando que o ALB está distribuindo o tráfego
✅

Se o hostname mudar entre as requisições, o Load Balancer está funcionando corretamente.

💡

E no ECS? O mesmo ALB e Target Group funcionam com ECS — na criação do Service, basta apontar o Load Balancer e o Target Group. O ECS registra e desregistra as tasks automaticamente conforme elas sobem e descem.

Parte 2 — CloudWatch + SNS
💡

O que é CloudWatch? O serviço de monitoramento da AWS. Ele coleta métricas de todos os serviços (CPU, memória, requisições, latência) e permite criar alarmes que disparam ações automáticas quando um valor ultrapassa um limite. O SNS (Simple Notification Service) é o serviço de notificações — ele envia e-mails, SMS ou aciona outras ações quando um alarme dispara.

Passo 5 — Criar um tópico SNS
05

Criar o tópico SNS para receber alertas

O canal de notificação que o CloudWatch vai usar

  • Procure por SNS (Simple Notification Service) → Tópicos → Criar tópico
  • Tipo: Padrão
  • Nome: alertas-aws
  • Clique em Criar tópico
  • Na página do tópico, clique em Criar assinatura
  • Protocolo: E-mail
  • Endpoint: seu e-mail
  • Clique em Criar assinatura
⚠️

Confirme a assinatura! A AWS envia um e-mail de confirmação. Clique no link "Confirm subscription" — sem isso, os alertas não chegam.

Passo 6 — Alarme de CPU alta
06

Alarme: CPU acima de 70%

Receber alerta quando uma instância estiver sobrecarregada

  • Procure por CloudWatch → Alarmes → Criar alarme
  • Clique em Selecionar métrica
  • Navegue até EC2 → Per instance Metrics desca até aparecer o docker-server, depois selecione apenas a métrica CPUUtilization dele
  • Clique em Selecionar métrica
CampoValor
EstatísticaMédia
Período5 minutos
CondiçãoMaior que
Limite70 (representa uma porcentagem)
Configuração adicional > Pontos para alarme2 de 2 (representa 10 min consecutivos). Clique em próximo
  • Em Notificação: selecione Em alarme → Enviar uma notificação para o seguinte tópico do SNS > selecionar um tópico SNS existente: selecione alertas-aws
  • Clique em Próximo. Nome do alarme: cpu-alta
  • Clique em Próximo. Clique em Criar alarme
💡

Por que 2 de 2? Exigir 2 períodos consecutivos evita falsos positivos — um pico momentâneo de CPU não vai disparar o alarme, apenas uma sobrecarga sustentada por 10 minutos.

Passo 7 — Alarme de requisições no Load Balancer
07

Alarme: requisições acima do limite

Receber alerta quando o tráfego estiver muito alto

  • Vá em CloudWatch → Alarmes → Criar alarme → Selecionar métrica
  • Navegue até ApplicationELB → Per ALB Metrics
  • Selecione a métrica RequestCount do seu alb
CampoValor
EstatísticaSoma
Período1 minuto
CondiçãoMaior que
Limite1000 (requisições/min)
  • Em Notificação: selecione Em alarme → selecione o tópico alertas-aws
  • Clique em Próximo. Em Nome do alarme: requisicoes-altas
  • Clique em Próximo. Clique em Criar alarme
💡

Ajuste o limite conforme sua aplicação. 1000 req/min é um exemplo — em produção, baseie esse valor no tráfego esperado da sua aplicação. Você pode visualizar o tráfego histórico nos gráficos do CloudWatch antes de definir o limite.

🧪 Teste — Ver métricas no CloudWatch

  • Vá em CloudWatch → Métricas → EC2 → Por instância
  • Selecione CPUUtilization das instâncias do ASG
  • Visualize o gráfico em tempo real — você verá as métricas atualizando a cada período
Passo 8 — Alarme de faturamento (CloudWatch)
08

Alarme de faturamento com CloudWatch

Ser avisado por e-mail quando a cobrança estimada da conta passar de um valor definido

1) Acessar o CloudWatch na região correta

  • Acesse o CloudWatch Console.

2) Criar um novo alarme

  • No menu à esquerda, clique em Alarmes.
  • Clique no botão Criar alarme.

3) Selecionar a métrica

  • Clique em Selecionar Métrica.
  • Procure e clique em Billing.
  • Clique em Total Estimated Charge.
  • Selecione a métrica: 
    EstimatedCharges
    e clique em Select metric.

4) Configurar o limite

  • Estatística: Maximum
  • Período: 6 hours
  • Tipo de limite: Estático
  • Sempre que EstimatedCharges for...: Maior
  • que... digite o valor limite, por exemplo: 
    30
    para ser avisado quando a cobrança estimada passar de 30 USD. Próximo!

5) Configurar a notificação (SNS)

  • Na seção de ações do alarme, selecione o estado In alarm.
  • Selecione seu tópico SNS.
  • Adicione seu endereço de e-mail como assinante desse tópico.
  • Confirme a assinatura clicando no link enviado para o seu e-mail.
  • Clique em Próximo. Informe o nome do alarme: custoalto. Próximo
  • Clique em Criar alarme

Depois de criar o alarme, sempre que a métrica EstimatedCharges ultrapassar o valor configurado, o alarme mudará para o estado ALARM e o CloudWatch enviará uma notificação por e-mail via SNS.

Parte 3 — IAM
💡

Observação: este laboratório não permite a criação de usuários, grupos e políticas do IAM. Em um ambiente com permissões completas, o procedimento seria: criar um grupo com permissões mínimas, criar o usuário (por exemplo, usuario-financeiro), adicioná-lo ao grupo e testar o acesso em uma janela anônima para confirmar que ele só consegue executar as ações permitidas.
